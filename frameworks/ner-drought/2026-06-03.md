# Niger drought — 2026-06-03 (development)

> In-development successor to the 2024-10-24 endorsed version. Not yet endorsed; no published framework doc. Trigger taken from the `iri-trend` branch and subject to change.
> The canonical trigger is the code at `code_ref`; this page explains it, it does not redefine it.

## Summary

In-development revision of the Niger drought AA framework. The core two-component trigger structure is retained from the 2024-10-24 endorsed version (IRI seasonal forecast + ENACTS SPI observational arm; two intervention windows; OR logic across arms and pairs). The key change under investigation on this branch is lowering the per-month forecast threshold from 35% to 30% to bring the historical activation probability within the CERF-required 3.5-year return period. An interactive marimo explorer deployed to GitHub Pages allows stakeholders to adjust thresholds and see the implied return periods and bad-year correlations. No new framework PDF has been published; this is pre-endorsement analysis.

## Method

IRI seasonal forecast probabilities of below-average JAS rainfall (OCHA Certification model, frequency slider at 35%) are exported from the IRI Maproom and assessed against a **rolling 10-year** historical reference window. For each month (Jan–Jun), the current year's value is compared against the *k*-th highest value in the preceding 10 years, where *k = ceil(pct/100 × n)*. This locks thresholds to real observed values (no interpolation). Two consecutive months must both exceed their respective thresholds for the forecast arm to fire in that month pair. Window 1 uses pairs Jan+Feb or Feb+Mar; Window 2 (forecast path) uses Mar+Apr, Apr+May, or May+Jun. The observational arm uses ENACTS MON Jun–Jul SPI (exported from the Maproom as the "Aug" column), compared against a **fixed** historical quantile from the full record.

Data exported from the IRI Maproom on 25 April 2026 (35% frequency slider). The app, README and `full_trigger.md` load `ds-aa-ner-drought/raw/iri/ner_maproom_export_2026-04-25_thresh35 - Sheet1.csv`; the threshold/RP analysis in `iri_trend.md` (the work that drives the 30% proposal) instead loads `ds-aa-ner-drought/raw/iri/ner_trigger_comparison_2025_newmodel_adjustment - raw.csv` (with `skiprows=3`). See Discrepancies — there is no single canonical input CSV on this branch.

## Trigger logic

- **Keys off:** IRI forecast probability of below-average JAS rainfall (OCHA Certification model) for months Jan–Jun; ENACTS MON Jun–Jul SPI for the observational arm.
- **Decision rule:** A year triggers if **either** arm fires (logical OR). The forecast arm fires if any two *consecutive* months (Jan+Feb, Feb+Mar, Mar+Apr, Apr+May, or May+Jun) both land in the top *X%* of their respective rolling 10-year reference windows. The observational arm fires if the Jun–Jul SPI is in the bottom *Y%* of the full historical record.
- **Proposed threshold:** forecast component X = 30% (down from 35% in the endorsed version); observational default Y = 15% (app default; inconsistent with the endorsed PDF's 35% — see Discrepancies).
- **Activation structure:** three window/path rows (Window 1 forecast, Window 2 forecast, Window 2 observational); within each arm OR across pairs; AND within a consecutive pair.
- **Calibration:** combined target return period ≈ 3.5 years (CERF minimum). The iri_trend.md analysis shows that at 35% per-month threshold the activation probability exceeds the 3.5-yr RP when computed over historical data from 1998 onwards; at 30% the RP is within acceptable range. The analysis also documents budget constants (W1: USD 5.25M, W2: USD 9.6M) for expected-spend calculations.
- **Authoritative source:** none — this is a development version; no published framework PDF exists for this trigger design. The repo on `iri-trend` is the only source.
- **Operated by:** the production trigger surface remains IRI Maproom 2 (operated with WFP/AGRHYMET/DMN/DNPGCA). This repo provides analysis and an explorer app; it is not the production system.

## Monitoring

The trigger explorer app is deployed as a WASM marimo notebook to a **dev-slot GitHub Pages deployment** (`https://ocha-dap.github.io/ds-aa-ner-drought/`) served from the `iri-trend` feature branch (`docs/` folder) — a stakeholder calibration tool, not a production deployment. It loads IRI data from a static CSV bundled in the export; locally it reads from Azure blob via `ocha-stratus`.

No automated monitoring pipeline runs from this dev branch. Companion monitoring is on the **old (endorsed) trigger**: IRI updates the Maproom mid-month (Jan–Aug) and reports trigger status against the endorsed 35% design. The 30% revision proposed here is not monitored anywhere live, and the operational trigger surface remains IRI Maproom 2 (operated with WFP/AGRHYMET/DMN/DNPGCA) — outside this repo.

## Key decisions & rationale

The central question motivating this branch is whether the 35% per-month forecast threshold (from the 2024-10-24 endorsed framework) continues to produce the intended ~3.5-year combined return period given the current IRI forecast data. The `iri_trend.md` analysis shows the IRI monthly forecast values have a detectable upward trend over time (increasing probability of below-normal forecasts), which inflates the historical trigger frequency at a fixed 35% threshold. Lowering to 30% restores the intended RP. The analysis also explores a detrending approach (beta-regression on the IRI series, back-transforming thresholds per year) as an alternative, but this was not adopted.

The rolling 10-year reference window (rather than an expanding window from 1991 or 1998) was the choice on this branch for the forecast arm, allowing the threshold to adapt as the IRI model and climate conditions evolve. The observational arm uses the full historical record (a fixed threshold), as there is less evidence of non-stationarity in the SPI series.

## Changes from previous version

The 2024-10-24 endorsed version uses a 35% per-month threshold for the forecast arm. The key proposed change here is **lowering the forecast threshold to 30%**, motivated by evidence that the 35% threshold over-fires relative to the 3.5-yr RP target when evaluated against data through 2025. The rolling 10-year reference window approach (already present in the explorer on the endorsed-era branch) is retained. The observational arm structure is unchanged; the unresolved 15%-vs-35% obs threshold conflict is carried forward from the endorsed version.

The trigger explorer app has been substantially reworked on this branch: Quarto/Plotly rendering replaced by a marimo WASM app deployed to GitHub Pages, with threshold sliders, return-period readouts, bad-year correlation analysis, and an interactive per-month detail chart.

## Open questions / known issues

- Should the forecast threshold be 30% or 35%? The iri_trend.md analysis supports 30%, but this has not been endorsed. What is the CERF/IRI/WFP sign-off process?
- The observational threshold remains 15% (app default) vs 35% (endorsed PDF) — which is operationally correct and should be adopted in this revision?
- Is the detrending approach (beta-regression, back-transformed thresholds) worth pursuing as a long-term solution instead of a fixed percentile cut, or is it too complex for operational use?
- The two-window structure (W1 "save the harvest" vs W2 "mitigate impact") is implicit in the month-pair groups but not explicitly configurable in the app — should it be surfaced as distinct sliders/packages?
- Budget constants in `iri_trend.md` (W1: USD 5.25M, W2: USD 9.6M) appear to be planning figures. Are these confirmed for the next endorsement?
- Is this revision intended to produce a new endorsed framework PDF, and what is the expected timeline?
