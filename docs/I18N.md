# I18N — multilingual public sites (EN/FR toggle, D86)

How the public AA site renders in more than one language, how to keep it that
way, how to reuse the mechanism on other public sites, and how to add a new
language (Spanish is the expected next one). The canonical French glossary —
mined from our own framework documents — is at the bottom.

Lives in **`scripts/site_i18n.py`**; used by the four AA-site generators
(`gen_aa_site.py`, `gen_public_site.py`, `gen_trigger_site.py`,
`gen_framework_pages.py`). Decision record: D86 in [DESIGN.md](DESIGN.md).

## The mechanism in one paragraph

Every generated page carries **all languages at once**; a client-side toggle
picks which one is visible. There is no per-language build, no duplicate pages,
no server: the generator emits each fixed UI string in both languages, and ~40
lines of shared JS/CSS swap them. The choice persists in `localStorage`
(`aa-lang`) — same-origin, so one click flips the whole site — and pages fire
an `aalang` event so anything JS-rendered (map legend, popovers, filter
widgets) re-renders itself. Shells (`gen_aa_site.py`) push the language into
their same-origin iframes.

## The pieces (all in `scripts/site_i18n.py`)

| Piece | What it does |
|---|---|
| `FR` | The single EN→FR table for every fixed UI string. EN string is the key. |
| `fr(en)` | Lookup with English fallback; every miss is collected and printed at generator exit as an `[i18n-missing]` warning — untranslated strings are **loud, not silent**. |
| `T(en[, fr])` | Emits `<span class="tr" data-en="…" data-fr="…">en</span>`. The client swaps `textContent`, so the DOM only ever contains ONE visible language — table **sorting and filtering** (which read `textContent`) keep working. Use for plain text. |
| `TB(en_html[, fr_html])` | Emits dual `<span class="lv lv-en">…</span><span class="lv lv-fr">…</span>` switched by CSS. Use when the string contains markup (links, `<b>`, `<code>`). |
| `JS_UI` | EN→FR dict serialized into pages whose JS builds text at runtime (map legend, popovers, filter placeholders). Client helper `tr8(key)` looks it up by the EN string. |
| `COUNTRY_FR`, `HAZARD_FR`, `STATUS_FR`, `MONTH_ABBR_FR`, `WINDOW_NAME_FR` | Controlled vocabularies (French exonyms, hazard labels, lifecycle statuses, months, common window names). |
| `LANG_CSS`, `LANG_JS`, `TOGGLE_HTML` | The shared plumbing each page embeds: the `.lv-*` switching CSS + toggle styling; `aaSetLang(l)` (sets `<html data-lang lang>`, swaps `.tr` spans, persists, updates `document.title` from `window.AA_TITLES`, dispatches `aalang`) + the load-time init (applies a stored choice on `DOMContentLoaded`, so placement in the page doesn't matter); the EN\|FR button pair. |

Dynamic data that feeds client-side rendering rides alongside its English in
the JSON payloads (e.g. map markers carry `country_fr`, `hazard_label_fr`,
`status_fr`), and the `aalang` listener rebuilds those DOM fragments — on the
map it also re-runs the callout layout, because label widths change.

## Rules of the road

- **Every new fixed UI string in a site generator gets a translation** — via
  the `FR` table (preferred: keeps everything in one place) or an explicit
  second argument to `T()`/`TB()` when the string is assembled from data
  (counts, dates). If you forget, the generator tells you at exit; a page never
  breaks, it just shows English.
- **Never translate data-derived free text** (trigger indicator/threshold
  prose parsed from KB pages, activation notes, AOI lists, window names quoted
  from documents). A baked translation of data goes silently stale the moment
  the data changes. It renders in its source language by design.
- **Numbers, currency, percentages keep ONE format across languages**
  (`$4.0M`, `36%`, `260,000`); only unit *words* translate (`12.3 years` →
  `12.3 ans`). The tables' client-side numeric sort parses cell text — a
  French decimal comma would break it. Revisit only with the sort/filter code.
- **Terminology is doc-sourced, not invented.** Before adding terms, check the
  glossary below; before adding a *language*, mine the framework docs in that
  language first (see the Spanish section).
- Language-independent behaviour must not key off display text: table columns
  carry `data-key` attributes (the EN column id) precisely so the filter/sort
  wiring survives a language switch. Follow that pattern for anything new.

## Wiring a new page or site

1. `import site_i18n as i18n; from site_i18n import T, TB` in the generator.
2. Wrap every fixed UI string: `T()` for plain text, `TB()` for markup.
3. Embed `{i18n.LANG_CSS}` in the page's `<style>`, `{i18n.LANG_JS}` in a
   `<script>`, `{i18n.TOGGLE_HTML}` in the header. Set
   `window.AA_TITLES = {en: …, fr: …}` for the tab title.
4. If the page renders text from JS, serialize `i18n.JS_UI` (see
   `gen_public_site.py`: `var UI = {…}; tr8()`), and add an `aalang` listener
   that rebuilds whatever JS drew.
5. `<select>` options can't hold spans — give options `data-en`/`data-fr` and
   swap `textContent` in an `aalang` listener (see `gen_trigger_site.py`'s
   framework picker).
6. Run the generator; fix any `[i18n-missing]` warnings.

A *different* site (another repo) can copy `site_i18n.py` wholesale — it has
no dependencies beyond the stdlib. If several repos end up carrying copies,
consider promoting it to `ds-toolkit`; until then, one file per site is fine.

## Adding a language (e.g. Spanish)

The mechanism is two-language today only in its data attributes, not in its
design. The checklist:

1. **Glossary first.** Mine the Spanish framework docs the same way the French
   glossary below was built: `raw/frameworks/` extracts for
   **lac-dry-corridor** (GTM/HND/SLV), **cub-hurricanes**, **nic-drought**
   have Spanish bodies; look for bilingual executive summaries giving exact
   EN↔ES term pairs, prefer the most frequent/official variant, and record
   sources + competing variants. Add the resulting table to this page.
2. In `site_i18n.py`: add `ES` (string table), `COUNTRY_ES`, `HAZARD_ES`,
   `STATUS_ES`, `MONTH_ABBR_ES`, `JS_UI_ES`, and have `T()`/`TB()` emit the
   extra `data-es` / `.lv-es` variant. The cleanest shape is to generalize to
   a `LANGS = {"fr": {...}, "es": {...}}` registry and loop, rather than a
   third copy of every helper.
3. `LANG_CSS`: add the `.lv-es` visibility rules; `TOGGLE_HTML`: third button;
   `LANG_JS` already reads `data-<lang>` generically.
4. Per-page JS: marker payloads gain `*_es` fields; `tr8()` picks the dict for
   the active language; `AA_TITLES` gains `es`.
5. `<html lang>` and any hard-coded `en/fr` conditionals — grep for
   `data-lang` and `'fr'` in the generators.
6. Regenerate, watch for `[i18n-missing]`, and click through all three views
   in the new language (the map layout especially — longer labels can change
   callout collision behaviour; that's why DR Congo's callout direction moved
   for French).

## French glossary (doc-sourced)

Mined 2026-07-21 from `raw/frameworks/` full-text extracts. Anchor sources:
**TCD-D** `tcd-drought/2025-03-03` and **MDG** `mdg-storms/2024-12-13` have
fully bilingual executive summaries (exact EN↔FR pairs); corroborated across
`tcd-flooding`, `ner-drought/-flooding`, `bfa-drought/-flooding`,
`cod-infectious-disease`, `hti-hurricanes`, `mrt-drought`.

### Conventions

- Months lowercase: *janvier … décembre*; accents matter (*août*, *février*).
- Compact ranges with a dash (*juin–septembre*), prose with *de … à*.
- French typography in UI strings: curly apostrophe (’), space before `:` and `;`.
